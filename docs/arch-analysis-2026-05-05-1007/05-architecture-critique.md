# Architecture Critique — Keisei

## Verdict

Keisei is a structurally sound system with one large, well-localised abscess. The boundaries (PyO3, SQLite, WebSocket) are the right ones, sized for the workload, and respected by the import graph. The Rust core, the typed config layer, and the WS hardening are above the bar for a post-reset rebuild. The league/tournament subsystem (C3) is not — it carries two parallel Elo-recording paths, a non-atomic match-result write, and a dense cluster of P1/P2 concurrency bugs that is incompatible with the system's own correctness goals. The right work is not a top-to-bottom refactor; it is to consolidate C3 onto a single execution path, atomise the result write, and decompose the two foundational single-file modules (`db.py`, `katago_loop.py`) before the next feature lands on top of them. The rebuild context is the explanation for how this got here; it is not a reason to grade it down.

## Methodology

I read the final report, the full subsystem catalog (20 entries), the discovery findings, the validation report, and skimmed `03-diagrams.md`. I did not re-read primary source code; every claim below cites a catalog entry or a file:line that the catalog or validation report verified. Sharper critique would require running `_record_match_result` under concurrent traffic, reading the bodies of `concurrent_matches.py` and `tournament_runner.py:_record_result` end-to-end, and a coverage map against C3.

## Strengths That Are Load-Bearing

1. **Pure-Rust core with zero runtime dependencies.** `shogi-core/Cargo.toml` lists only criterion as `[dev-dependencies]`; the 11 modules import `std` only (catalog A). This isolates UB risk to `shogi-gym` and keeps `cargo test -p shogi-core` fast and self-contained. The make/unmake symmetry with explicit `UndoInfo` carrying `prev_hash`/`prev_attack_map`/`was_in_check` (`game.rs:19`) and the niche-optimised `Piece(NonZeroU8)` so `Option<Piece>` is one byte (`piece.rs:12`, verified by `test_option_piece_is_one_byte`) are the kind of detail that signals real engineering, not aspiration.
2. **Centralised typed configuration with cross-registry validation.** Seventeen frozen dataclasses; `algorithm` and `architecture` validated against `VALID_ALGORITHMS`/`VALID_ARCHITECTURES` imported from training registries (`config.py:14-18, 582, 639`); legacy section names raise on load (`config.py:654-666`); paths resolve relative to the TOML (`config.py:594, 625`); `enabled=false` collapses `LeagueConfig` to `None` so consumers' `if config.league is not None` short-circuits cleanly (`config.py:730-733`). One source of truth, no globals scattered across modules.
3. **DB-as-message-bus discipline holds in practice.** WebUI never opens SQLite (catalog cross-cutting, line 397). Server never opens a socket to training. The only out-of-bucket import from training to server/showcase is `model_registry.{build_model, get_model_contract, get_obs_channels}` for inference-time architecture lookup. Cross-bucket coupling is limited to the FFI surface (9 PyClasses), the 21-table schema, and 12 + 4 WS messages. Validation Boundary Ownership Audit found no leaks across 20 entries.
4. **WS layer has been hardened against real failure modes, not theoretical ones.** Per-connection `asyncio.Lock` wraps every send, with the cited library line that motivated it (`app.py:84-102`); manual host re-check inside `ws_endpoint` because `BaseHTTPMiddleware` doesn't filter WS scope (`app.py:226`); `except*` PEP 654 exception-group flattening (`app.py:74, 250-263`) so per-task tracebacks are not lost. These are concrete fixes for concrete bugs, not generic robustness.
5. **`shogi-gym` two-phase step contract is the correct decomposition.** Validate under GIL, then `py.allow_threads(|| apply_moves(...))` (`vec_env.rs:705`); rayon `par_iter` above `PARALLEL_THRESHOLD = 64`; per-env `catch_unwind` with sentinel reset on panic (`vec_env.rs:462`). The closed P2 `keisei-cdf80418a1` confirms catch_unwind was added 2026-04-03 — not aspirational hardening, applied hardening.

## Material Weaknesses

1. **Critical — Non-atomic match-result recording.** `LeagueTournament._record_match_result` (`tournament.py:352-460`) issues `store.record_result`, two `store.update_elo` calls, and `role_elo_tracker.update_from_result` as **separate transactions on `OpponentStore`**. Filigree `keisei-fa604bad63` (P1). A crash between writes leaves Elo split across half-applied state. Catalog C3 confirms this is real, not theoretical. **What good looks like:** one `OpponentStore` transaction context manager wrapping all four writes; on crash either everything or nothing lands. The store already exposes `transaction()` per `opponent_store.py:334`; the fix is mechanical.
2. **Critical — Two parallel match-recording paths.** `tournament._record_match_result` and `tournament_runner._record_result` reimplement the same Elo + DB write sequence (catalog C3 cross-cutting, line 290). Sidecar lost-update race tracked as `keisei-ea85c3d5b5`; dispatcher's `PriorityScorer` never sees sidecar completions (`keisei-6cb0990f53`); `check_round_completion` not idempotent (`keisei-9cae4c09b4`). The final report calls this "the single biggest structural debt." **That judgment is correct.** Two paths means two places to fix every bug, two places where divergence drifts silently, and the integration point (`katago_loop.py:672`) selects between them at config-load time so a single training run only exercises one. **What good looks like:** one path. Sidecar is the right endpoint (process isolation, GPU contention isolated, restart semantics cleaner) — the in-process `LeagueTournament` thread should be removed, not refactored alongside.
3. **High — `katago_loop.py` is the next regression cluster.** 1,989 LoC in one class, directly importing 14 in-bucket modules (validation report; catalog C1b). Loop, opponent rotation, league bookkeeping, snapshotting, DB writes, and `keisei-train` CLI colocated. One ad-hoc raw-SQL `UPDATE training_state.total_epochs` (`katago_loop.py:843-848`) bypasses the `db.py` helper layer. **What good looks like:** extract `LeagueOrchestration` (the league wiring around `TieredPool`/`LeagueTournament`/`HistoricalGauntlet` — currently ~700 LoC inside the class), `CheckpointManager` (snapshotting/saving, currently shared with the `_save_*` helpers), and the CLI `main()`. Target: the `KataGoTrainingLoop` class < 800 LoC, exclusively the per-step learn/dispatch path. The 14 imports drop to ~7.
4. **High — `concurrent_matches.py` slot/partition/LRU bug cluster.** P1 `keisei-53eb4eb1f8` (slot reuse without partition reset leaks games), P1 `keisei-4b6c36cd2b` (`.cpu()` cleanup poisons shared LRU at `opponent_store.py:344`), plus P2s `keisei-bc58948f9f`/`f2189813df`/`08ccd20240` (catalog C3 Concerns). These are not stylistic — they are correctness bugs in the path that produces all training-time match data. **What good looks like:** explicit `_MatchSlot` lifecycle state machine with partition reset on every `reset_for_pairing`; immutable model handles in the LRU (load returns a frozen wrapper that disallows in-place `.cpu()`); a `tests/test_concurrent_matches.py` that exercises slot reuse across pairings.
5. **High — `historical_gauntlet.run_gauntlet` tuple-unpacks a dataclass.** `MatchOutcome` is a dataclass (`match_utils.py:21`), not a tuple — every gauntlet match is silently treated as failure (`keisei-4509042dd1`, P1). Catalog C3 confirms. This is a milestone-evaluation regression that produces zero signal where the system claims to produce its primary external benchmark. **What good looks like:** fix the unpacking, add a regression test that asserts non-zero wins on a hand-crafted always-win opponent.
6. **High — `db.py` is over the line.** 1,158 LoC, 21 `CREATE TABLE` statements, eight migration functions, read/write helpers for ≥12 entity families (catalog E2). The catalog calls it "approaching maintenance threshold." That is too charitable — touching one entity now requires reading or scrolling past the others. **What good looks like:** `keisei/db/__init__.py` exposing `_connect`, `init_db`, `wal_checkpoint` and the migration runner; per-entity submodules (`metrics.py`, `snapshots.py`, `training_state.py`, `league.py`, `head_to_head.py`, `historical.py`, `gauntlet.py`, `tournament.py`, `game_features.py`, `style_profiles.py`, `showcase.py`, `tournament_queue.py`). The migration registry stays in one place; the read/write helpers live with their tables. Schema constants and the v1→v8 chain are stable enough to split now.
7. **Medium — `config.py` at 759 LoC is on the line, not over it.** Seventeen frozen dataclasses with `__post_init__` validation (catalog E1). Still readable end-to-end; legacy/deprecation handling and `[league.active]` sugar account for ~150 LoC that are inherently single-place. Decompose only on the next major schema change. **What good looks like:** if/when split, group by bucket — `config/training.py`, `config/league.py` (10 dataclasses), `config/distributed.py`, `config/app.py` (composition + `load_config`). Not yet.
8. **Medium — `vec_env.rs` `unsafe`-by-convention surface is acceptable but under-defended.** `unsafe impl Send/Sync for SendPtr<T>` (`vec_env.rs:66-67`) and two large `unsafe` blocks (`:348-458` happy path, `:480-539` panic recovery) write through pointers under a disjoint-index invariant enforced only by `debug_assert` (catalog B Concerns). The four mode-tag dispatch sites must stay in sync when adding a new mode (`vec_env.rs:128-137`); only exhaustive `match` enforces this. **The throughput justifies the unsafety** — rayon-parallel per-env step at `PARALLEL_THRESHOLD = 64` is the hot path and pre-allocated flat buffers are the right shape for it. **What good looks like:** wrap the unsafe writes in a typed `DisjointBufferWriter<T>` with a `release` mode that retains the `debug_assert`s, plus a release-mode property test (`proptest`) that asserts disjointness across N envs × M steps. Add a single `TODO: KEEP IN SYNC` block above all four dispatch sites with a compile-time assertion. The closed P2 `keisei-cdf80418a1` proves the team can land Rust hardening; this is the next item in that queue, not a reason to redesign.
9. **Medium — Server-side init undercounts plus silent fingerprint misses.** `total_episodes` truncated to first 500 metrics rows then advances cursor (`app.py:354`, `keisei-f11f3179f7`); style-profile fingerprint captures only `(checkpoint_id, status, primary_style)` so raw_metrics/percentile recomputes slip past (`keisei-8108b42644`); showcase status fingerprint misses speed-change diffs (`keisei-d26b243465`); `_handle_match_request` skips entry-id existence (`keisei-bcc71be6ef`); `gpus` key missing on non-zero `nvidia-smi` exit (`keisei-974d6aba11`). Roughly seven open server triage items in catalog F. Aggregate: the dashboard misreports state, but training is unaffected. **What good looks like:** a `Fingerprint = NamedTuple[...]` with a class-level invariant that every field that contributes to "should we re-push" is in the tuple, plus a unit test per fingerprint asserting equality changes when each field changes.
10. **Medium — Showcase sidecar liveness is on the same thread as the game loop.** `_run_game` (`runner.py:113`) blocks the heartbeat; long games silently mark sidecar offline (`keisei-6045789532`, catalog G). Plus `ModelCache` key omits checkpoint mtime (`inference.py:127`, `keisei-8c55b48bcc`) — same path serves stale weights after re-save. **What good looks like:** a separate heartbeat thread (`threading.Thread(daemon=True)`) writing every 10 s independent of game progress; LRU key includes `(entry_id, checkpoint_path, mtime_ns)`.
11. **Medium — `katago_ppo.py` validation gaps and `flush_timings` never called.** `KataGoPPOParams.__post_init__` validates lower bounds but not NaN or upper bounds (`keisei-cb7008ac73`); `MultiHeadValueAdapter` accepts negative `lambda_value`/`lambda_score` (`keisei-bef32b64a8`/`keisei-678359b7aa`) — gradient ascent occurs silently. `KataGoPPOAlgorithm.flush_timings()` is never called (`keisei-ca5e280cae`, P2 in-progress) — `select_actions` CUDA events accumulate unboundedly. **What good looks like:** explicit `validate_finite_positive(field)` helper in `__post_init__` for every learning-rate-shaped scalar and lambda; `flush_timings()` called once per epoch from the loop's existing per-epoch hook.
12. **Low — Svelte 4 → 5 migration is correctly rated P4.** Every component uses `export let`, reactive `$:` blocks pervasive, `createEventDispatcher` in `MoveLog`, `before/afterUpdate` in `MoveLog`/`MetricsChart`/`PlayerCard` (catalog cross-cutting Svelte 5 section, plus H1/H2/H3 Concerns). Tracked as `keisei-9b1171d032`/`a5fe9f710e`/`975949c0b3`/`a1622bc4cf`. **The P4 rating is correct.** UI works; Svelte 4 is supported; there is no user-facing bug. The migration is mechanical (mostly a codemod), large in surface area, and gets cheaper if you do it once after the next round of UI feature stabilisation, not now while components are still moving. Holding it at P4 is right.

## Architectural Themes

1. **Convention, not mechanism.** Five distinct mechanisms in the system rest on documented invariants enforced at most by `debug_assert` or by import-graph hygiene: (a) `vec_env.rs` disjoint-index writes (W8); (b) DB single-writer-per-table (catalog E2 confidence note — `check_same_thread=False`, no pool); (c) lazy `shogi_gym` imports as a "don't load Rust ext at import time" seam in 7 modules (catalog cross-cutting); (d) `OpponentStore._pinned` as in-memory pin set, lost on restart (`opponent_store.py:341`, `keisei-76cc7fdc85`); (e) WebUI fingerprint-based dedup for incremental WS pushes (W9). Each is fine in isolation. The pattern is the concern: **the system has accreted enforcement-by-doc-comment as its default form of invariant.** Systemic fix is not to mechanise all of them — it is to mechanise the ones with correctness consequences (vec_env unsafe writes; DB writer arbitration once the league sidecar is wired in) and consciously accept the rest as cheap convention.
2. **Dual paths instead of one.** Three separate places: in-process `LeagueTournament` vs sidecar `tournament_runner` (W2); `_record_match_result` reimplemented in both (W1, W2); training writes vs showcase writes to different DB regions but with the same `_connect` and same WAL discipline (catalog G "Interaction with db.py vs own tables"). The first two are the same problem. The third is OK because the writes don't overlap. **The systemic fix is finishing the migration to the sidecar, not adding a third path or refactoring both in parallel.** Every C3 P1/P2 in the catalog (~13 issues) gets cheaper to fix once there is one path; many become moot.
3. **Single-file foundations growing under their own gravity.** `katago_loop.py` 1,989, `db.py` 1,158, `config.py` 759, `opponent_store.py` 1,324, `vec_env.rs` 1,996. The first two are over the maintenance line (W3, W6); the third is on it (W7); the last two are domain-justified (one class, one PyO3 surface). **The systemic fix is splitting `db.py` and `katago_loop.py` first** — they are not domain monoliths, they are accumulations. `opponent_store.py` should not be split, it is one cohesive transactional surface; `vec_env.rs` should not be split, it is one PyClass; `config.py` waits.
4. **Asymmetric bucket shapes are mostly justified.** Training bucket has ~30 modules; server is one file (658 LoC); WebUI is 29 components but no router. (a) Training is genuinely complex — PPO + multi-tier league + concurrent matches + sidecar + SL — so 30 modules is honest. (b) The single-file server is correct: 4 routes, one WS, one factory; splitting would be ceremony. (c) The WebUI's lack of a router is correct: four mutually exclusive top-level views, switched by `{#if $activeTab}` (catalog H1). The asymmetry is workload-driven, not accidental. The one place this breaks is **`App.svelte` doubling as the Training tab layout** (catalog H1 Concerns) — extract `TrainingView.svelte`, then the four-view symmetry is real.
5. **The validation report's confidence calibration is the right kind of honest, and surfaces the actual depth limit of this critique.** Per the catalog: `OpponentStore` read 1-360 of 1,324; `LeagueTournament` 1-460 of 658; `tournament_runner._record_result` not read; `concurrent_matches.py` slot-reuse fault paths relied on filigree IDs. The catalog cannot distinguish "this bug is real and minor" from "this bug is real and the surrounding code is also broken in ways no one has filed yet." The implication: **C3 is at least as bad as catalogued, possibly worse.** That strengthens, not weakens, the consolidation argument.

## Decision Points the Architect Must Make

1. **League execution: keep dual paths / consolidate to sidecar / consolidate to in-process.** **Recommend: consolidate to sidecar.** Reasons: process isolation contains GPU and panic blast radius (the in-process `LeagueTournament` thread shares `katago_loop.py`'s process); restart semantics are cleaner (sidecar already has worker heartbeat at `tournament_worker_heartbeat`); the sidecar is the newer code path with the post-2026-04-01 design assumptions (lossy/idempotent claim queue, atomic `BEGIN IMMEDIATE` UPDATE per `tournament_queue.py`). Cost: rewriting one path's callers in `katago_loop.py`; tooling for the sidecar process supervision (or relying on the existing `_startup_cleanup` showcase pattern as a template). The dispatcher idempotency bug (`keisei-9cae4c09b4`) and `PriorityScorer`-doesn't-see-sidecar-completions (`keisei-6cb0990f53`) become forced fixes, not background drift.
2. **`_record_match_result` atomicity: fix transactionally first, or after consolidation.** **Recommend: transactionally first, then consolidate.** The transactional fix is small and unblocks every other P1 in C3 from compounding partial-state symptoms. Wait for consolidation and you ship known-broken atomicity for the duration of the migration. Wrap all four writes in one `OpponentStore.transaction()` block; ship it; then consolidate paths.
3. **`db.py` decomposition: split now / split later / leave it.** **Recommend: split now**, before the next entity table lands. The migration chain is stable through v8; the per-entity write helpers are independent. Cost is ~1 day, low risk. Leaving it accretes another ~150-200 LoC per quarter at current pace.
4. **`katago_loop.py` decomposition: split now / split later / leave it.** **Recommend: split after C3 consolidation.** Reason: half the dependencies of `katago_loop.py` are league-wiring; consolidating C3 changes the import graph for that half. Split after, not before, or you'll do it twice.
5. **`vec_env.rs` unsafe surface: add property tests / wrap in typed writer / ignore.** **Recommend: add property tests, do not redesign.** The throughput justifies the unsafety; the production stability justifies leaving the structure. A `proptest` over disjointness across N×M, plus the existing `debug_assert`, is the cheapest defence.
6. **Showcase heartbeat: separate thread / async task / accept the bug.** **Recommend: separate thread.** Sidecar already uses `threading.Thread` patterns; one more daemon thread for heartbeat is trivial. The bug is real (`keisei-6045789532`) and the fix is small.
7. **WebUI Svelte 5 migration: do it now / wait / never.** **Recommend: wait.** P4 is the right rating. Do it after the next round of UI feature stabilisation when components are settled; do it as one codemod-driven PR, not a creeping migration; do not let new components land in mixed Svelte-5 idiom before the codemod runs.
8. **Test gap analysis: now / after C3 consolidation / never.** **Recommend: after C3 consolidation.** Coverage signal today would mostly say "C3 is undertested" — which the filigree backlog already says. After consolidation, a coverage map identifies the real gaps in the consolidated path; before consolidation, it identifies gaps in code that's about to be deleted.

## What I Would NOT Touch

- **`shogi-core` zero-deps posture.** Eleven modules, `std`-only, criterion as `[dev-dependencies]` (catalog A). Do not add a single runtime dep. The Rust core is the most stable part of the system precisely because of this.
- **The 9-PyClass FFI surface.** Nine classes, exact contract documented (catalog B "FFI Exported Surface"); seven Python consumers all use `observation_mode="katago"` + `action_mode="spatial"`; ZST static asserts at compile time. Adding to it should require explicit ADR; do not bend the surface to accommodate one caller.
- **The WS protocol's hardening.** Per-connection `asyncio.Lock`, host re-check, `except*` flattening, fingerprint-based dedup, timeouts. The fingerprint *contents* are buggy (W9); the *mechanism* is right.
- **The 21-table SQLite schema.** v8 with eight chained migrations including v4→v5's filtered backfill (`db.py:99-103`); single-row tables with `CHECK (id = 1)`; partial UNIQUE INDEX `idx_showcase_queue_one_running`; canonical H2H ordering. The schema discipline is real and should not be relaxed for convenience.
- **Typed config layer.** Frozen dataclasses with `__post_init__`; legacy section rejection; cross-registry validation; path resolution. Touch only to add new fields with the same idiom.
- **`OpponentStore` as a single file.** 1,324 LoC, but it is one transactional surface over many DB tables (composite + 4 role-specific Elo columns; in-memory model LRU; pin set; `RLock` reentrance contract documented at `tier_managers.py:102-104`). Splitting it would create artificial seams across what is genuinely one mutex.
- **The DB-as-message-bus discipline.** WebUI does not open SQLite. Server does not open a socket to training. Out-of-bucket import budget is one (`model_registry` from showcase). Do not relax this for convenience.
- **`uplot` as the only chart library.** Two call sites (`MetricsChart`, `WinProbGraph`); one CSS import (catalog H5). Adding a second chart library would be a regression.

## Confidence and Caveats

**Per-claim confidence:**
- High: every weakness W1-W12, the three integration boundaries, the dual-path diagnosis, the catalog's LoC and validation-pass numbers — all directly cited from catalog entries that the validation report cross-checked.
- Medium: the recommendation to consolidate onto the sidecar rather than in-process is a judgment call; both paths are wired and the sidecar is newer, but I have not run either under load. The claim that `db.py` is "over the line" rests on the catalog's LoC + entity-family count, not on a maintenance friction measurement.
- Lower: the systemic-fix shapes I propose for W3, W6, W8, W10 are defensible architectural moves but are not the only correct shapes — different splits could plausibly be justified.

**Information gaps inherited from validation:**
- `tournament_runner._record_result` body was not read by the catalog agent (catalog C3 confidence note). My W1/W2 critique relies on the symptom description in `keisei-ea85c3d5b5`/`keisei-fa604bad63`, not on a side-by-side diff.
- `concurrent_matches.py` slot-reuse fault paths relied on filigree IDs; the catalog states "Low" confidence on this. W4 inherits that limit.
- `LeagueTournament` read 1-460 of 658; `OpponentStore` 1-360 of 1,324. Inner-method semantics not exhaustively traced. The "one transaction context manager" fix for W1 assumes `OpponentStore.transaction()` is reentrant in the way `tier_managers.py:102-104` documents; if that documentation is stale, the fix is larger.
- WebUI LoC not aggregated; my WebUI claims are by file count and per-component reads.
- Static-only analysis. No profiling, no runtime trace, no fuzzing. Performance claims (e.g. that the fix shapes are cheap) are structural arguments, not measurements.

**Caveats:**
- This is an architectural critique, not a security or correctness audit. I did not threat-model the WS surface or formally verify the soundness of the `unsafe` blocks; the final report flagged both as appropriate hand-offs to `ordis-security-architect` and `ordis-quality-engineering` respectively.
- Rebuild context (repo reset 2026-04-01, sidecar partially landed alongside legacy in-process tournament) is the explanation for the C3 mess; it is not a reason to grade C3 less harshly. I have explicitly not done that.
- Filigree priority labels were not returning P1/P2 in standard query format at catalog construction time (catalog cross-reference line 1006). My inherited priority signal is therefore weaker than ideal; cited priorities reflect the catalog agent's reading of the issue body, not a label query.
